## Project Overview

Microsoft Intelligent Predictive Maintenance Hackathon — a multi-agent AI system for a tire manufacturing factory. Five agents form a sequential pipeline: Anomaly Classification → Fault Diagnosis → Repair Planner → Maintenance Scheduler → Parts Ordering. Built with Microsoft Agent Framework, Azure AI Foundry, and Azure services.

## Architecture

- **Challenge 0**: Environment setup, Azure resource deployment, Cosmos DB seeding
- **Challenge 1** (Python): Anomaly Classification Agent (function tools + Cosmos DB) and Fault Diagnosis Agent (MCP tools + Foundry IQ knowledge base)
- **Challenge 2** (C#/.NET): Repair Planner Agent using Foundry Agents SDK — built via GitHub Copilot @agentplanning agent
- **Challenge 3** (Python): Maintenance Scheduler + Parts Ordering agents with persistent memory (Cosmos DB chat history) and OpenTelemetry observability
- **Challenge 4** (Python + C#): End-to-end orchestration via Aspire — FastAPI exposes Challenge 3 agents as A2A endpoints, .NET FactoryWorkflow orchestrates the full pipeline, Vite React frontend

## Commands

```bash
# Environment setup (run from challenge-0/)
./get-keys.sh --resource-group <RG_NAME>    # Extracts Azure keys into ../.env
export $(cat ../.env | xargs)                # Load env vars (re-run after each new terminal)
./seed-data.sh                               # Seeds Cosmos DB + Blob Storage + APIM APIs

# Python dependencies (from repo root)
python3 -m venv .venv
source .venv/bin/activate
pip install --pre -r requirements.txt

# Run Challenge 1 agents (from challenge-1/)
python agents/anomaly_classification_agent.py
python agents/fault_diagnosis_agent.py

# Run Challenge 3 batch (from challenge-3/)
python run-batch.py                          # Processes 5 work orders through scheduler + ordering agents

# Run Challenge 4 orchestration (from repo root)
dotnet run --project challenge-4/agent-workflow/dotnetworkflow/FactoryWorkflow.csproj
# Or via Aspire (starts FastAPI + .NET workflow + frontend):
# Aspire apphost is at challenge-4/agent-workflow/apphost.cs

# .NET build
dotnet build factory-hack.sln
```

## Cosmos DB Data Model (FactoryOpsDB)

9 containers: Machines (`/type`), Thresholds (`/machineType`), Telemetry (`/machineId`), KnowledgeBase (`/machineType`), PartsInventory (`/category`), Technicians (`/department`), WorkOrders (`/status`), MaintenanceHistory (`/machineId`), MaintenanceWindows (`/isAvailable`).

All 5 machines have pre-seeded warning telemetry for agent testing.

## Critical Constraints

- **Do NOT reference** `example-solution/`, `solutions/`, or any `*solution*` folders — treat as off-limits
- **Challenge 2 (.NET)**: Use Foundry Agents SDK (`Azure.AI.Projects` + `Microsoft.Agents.AI`), never `Azure.AI.Inference`/`ChatCompletionsClient`
- **Pinned NuGet versions** (Challenge 2): `Azure.AI.Projects` 1.2.0-beta.5, `Microsoft.Agents.AI` 1.0.0-preview.260108.1, `Microsoft.Azure.Cosmos` 3.56.0, `Azure.Identity` 1.17.1
- **Fault→Skills/Parts mappings** in Challenge 2 are hardcoded in-memory dictionaries in `FaultMappingService` (see `.github/agents/agentplanning.agent.md` for values)
- Python requires `--pre` flag for pip install (agent-framework packages are pre-release)
- Challenge 4 Python requires Python >=3.13, .NET targets net10.0

## Agent Patterns

- **Function tools** (Challenge 1 anomaly): Agent defines Python functions decorated as tools, calls Cosmos DB directly
- **MCP tools** (Challenge 1 fault diagnosis): Agent uses MCP server endpoints (APIM proxy to Cosmos) + Foundry IQ knowledge sources
- **A2A protocol** (Challenge 3→4): Python agents wrapped as Starlette A2A apps, called by .NET orchestrator over HTTP
- **Persistent memory** (Challenge 3): Chat history stored in Cosmos DB, loaded per machine context
